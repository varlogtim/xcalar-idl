Using batch dataflows

A Batch Dataflow is a modular algorithm resulting from your data modeling to produce a set of data. The intent of a batch dataflow is to apply defined modeling actions to different sets of data or to a full corpus of data at scale. This section describes tasks related to batch dataflows except the following tasks:

  • Scheduling, which is described in Scheduling batch dataflows.
  • Customizing dataset import, modeling operations and export, which is described in Customization of dataflow operations.

Displaying batch dataflows

Follow these steps to display batch dataflows:

  1. Click to display a list of batch dataflows created by all users.
  2. Click the name of a batch dataflow in the list to display its dataflow graph.

Result of running a batch dataflow

The result of running a batch dataflow can be exported to an export target or as a table in the active worksheet.

Exporting to an export target

By default, the batch dataflow exports .csv format data to a folder in the storage location defined as the default export target. The name of this folder is displayed below the batch dataflow graph's export icon. The data generated by the dataflow is saved as one or multiple files in this folder. By default, this folder's name is the same as the name of the results table in the original dataflow.

EXAMPLE: Suppose you create a batch dataflow from the dataflow for a table named carriers#h935. In the batch dataflow, the last icon represents the export folder, which is named export-carriers#h935. If you run this batch dataflow, the output is saved to csv files located in export-carriers#h935. The folder is created at the default export target. For information about the meaning of an export target, see Exporting a table.

Exporting to a table

The batch dataflow can export the result of a batch dataflow run to a table containing selected columns. The newly created table is a starting point for additional modeling. The table is added to the current workbook, and not to the workbook where you created the dataflow. All columns in the newly created table are derived columns, with no lineage and no column prefix.

Understanding other export locations

To export data from a batch dataflow to a location other than the default folder, customize your export operation, by specifying one of the following export locations:

  • Export target: You can specify the name of the export target and/or the folder that will contain the batch dataflow results. The target can be the default target or a custom target. After you make your selection, a custom folder name replaces the default folder name below the last icon of the dataflow graph.

    IMPORTANT:The name of the export file must not be a duplicate. For example, if running the batch dataflow for the first time creates a file named export-airlines1#L811.csv, re-running the batch dataflow fails because the dataflow attempts to create an export file with the same name. To avoid duplicating names, before re-running a dataflow, either rename the existing export file (for example, to export-airlines00#L811.csv) or customize the export file name.

  • Table in the current worksheet: The batch dataflow creates a table under a name you specify. There are two requirements for exporting to a table:

    • You must run the dataflow from the dataflow graph. If you run the dataflow according to a schedule, the batch dataflow always exports the result as file(s).
    • There is only one dataflow in the batch dataflow. If a batch dataflow graph shows multiple dataflows, each dataflow can export only to an export target, not a table.

Running a batch dataflow created by Xcalar in modeling mode

You can create a batch dataflow on a cluster in Xcalar in modeling mode, download it to a dataflow file, and then upload it to a cluster running Xcalar in operational mode. For information about moving a batch dataflow from one cluster to another, see Downloading and uploading a batch dataflow.

Tasks before running a batch dataflow

Before running a batch dataflow created on another cluster, be sure to perform these tasks:

  • Verify that the data target and data source path are correct. Be aware that the cluster that creates the batch dataflow and the cluster to which the batch dataflow is uploaded might have different data targets.
  • Verify that the location displayed below the last icon of the dataflow graph is the correct. If you want the results to be saved in another place, you must create a customized operation to modify or add a parameter to the export operation.
  • Verify that the export operation is customized appropriately for your cluster.
  • If the batch dataflow contains parameters, verify that values are assigned to the parameters.

Understanding tables exported from batch dataflows

A results table exported to the current worksheet from a batch dataflow works in the same way as any other table. It has its own dataflow graph. The first icon of this dataflow graph, however, is not a dataset. It is an icon representing the exported table itself. From this dataflow, you can create another batch dataflow.

The following differences exist between the table created through a batch dataflow and other tables:

  • All columns are derived columns. Column names do not have a prefix.
  • From the table's dataflow, you cannot trace the lineage of the data back to the batch dataflow that created it. For example, you cannot determine the source of the data or whether the data type of a column has been changed.

A batch dataflow created from a table generated from another batch dataflow is called a cascading batch dataflow.

Creating a cascading batch dataflow

A cascading batch dataflow is a batch dataflow created from a table that is an export output from an existing batch dataflow. Creating a cascading batch dataflow enables you to extend the modeling steps in a batch dataflow.

For example, you can create a batch dataflow named airlines3Data that reads airlines data from a dataset, casts all columns to the appropriate data type, and filters a value in the DayOfWeek column. Suppose you or another user of the Xcalar cluster wants to add operations to this batch dataflow, follow these steps to create a cascading dataflow:

  1. Customize the export operation of the airlines3Data batch dataflow to export to a table, for example, named FilteredAirlinesData.
  2. Run the airlines3Data batch dataflow to create the FilteredAirlinesData table in the current workbook.
  3. Locate FilteredAirlinesData in the worksheet.
  4. Perform the desired operations on FilteredAirlinesData. For example, you can filter rows with AirTime that is greater than 120.
  5. Display the dataflow graph for FilteredAirlinesData and create a batch dataflow from it. This newly created batch dataflow is a cascading batch dataflow.

Exporting from a batch dataflow to a specified folder or table

You can see the export folder name below the last icon of the batch dataflow graph. You can change the folder by following these steps:

  1. Click the last icon in the batch dataflow graph. Then click Customize operation in the pop-up menu.

    Go to step 2 if you want to export to a folder. Go to step 3 if you want to export to a table.

  2. In the Customize Export Operation modal window, follow one of these steps to specify the export folder:

    • Enter an export folder name that does not conflict with any existing folder name at the export target.
    • Specify an export target that is different from the target that contains the existing export folder.

    Both the export folder name and the export target can be parameterized. For example, you can create a parameter named suffix and insert the parameter into the export folder name or export target name. For more information about parameterization, see Customization of dataflow operations.

    NOTE: The parameterized target name must refer to a target that has been created. For example, if the parameterized target name is myTarget1 but no target under that name exists, an error message is displayed.

    Go to step 4.

  3. In the Customize Export Operation modal window, select Export as a Xcalar table and specify a table name.

    The following list describes the naming conventions:
    • Length must not exceed 255 characters.
    • Characters must be A-Z, a-z, 0-9, or underscore (_).
    • First character must be a letter.
    • Letters are displayed in uppercase, but you can enter them in uppercase or lowercase.

  4. Click SAVE.

  5. In the dataflow graph, the name under the last icon is changed to the export folder or table name.

    IMPORTANT: If you choose to parameterize the export folder name or target name, be sure to use a parameter created by you. Do not use a system parameter, such as the N parameter, which is reserved for scheduled batch dataflows. If the export folder or target name contains a system parameter, a warning notifies you that the parameter is disregarded. If you click CONFIRM to proceed, the batch dataflow is run with no parameterization.

Running a batch dataflow

Click to run the batch dataflow. When the batch dataflow is in progress, the icon turns into a spinner. You can click the spinner to terminate the batch dataflow.

The following list describes how the batch dataflow graph changes when a batch dataflow is in progress or is finished:

  • When the batch dataflow is in progress, there is a tooltip showing the status of each table that has been created or is being created.
  • For each table being created, the icon is yellow, and a progress bar is displayed.
  • After a table is created, the dataset and table icons turn green. You can hover over each icon to display the time taken to complete the operation and the number of rows in the table.
  • If the batch dataflow fails to complete, a red dataset or table icon is displayed to show where the problem occurs.

Downloading and uploading a batch dataflow

You can download a batch dataflow to your client computer, the system where the browser for Xcalar Design is running. Initiating the download process creates a .tar.gz fileâ€”a compressed directory containing dataflowinfo.json and UDFs that are required for the dataflow. The dataflowinfo.json file contains the dataflow written in the Xcalar query language.

The .tar.gz file is also called a dataflow file. After the dataflow file is downloaded, you can upload it to any Xcalar cluster from your client computer with a Xcalar Design browser session.

The download location is determined by a browser setting. For example, in your Chrome browser, display the advanced settings and then go to the Downloads section to specify a path for storing all downloaded files.

Understanding the dataflow file contents

The dataflow file is a portable representation of your data model. It contains all the operations and customizations, which constitute the algorithm designed through your modeling process. It does not retain the values that pertain to the actual running of the dataflow. You must supply those values after uploading the dataflow. The following sections describe specifically the types of information that are included or excluded in a dataflow file.

IMPORTANT: Do not edit the dataflow file; changing its contents might cause it not to run properly.

Information included in the dataflow file

The dataflow file includes the parameters used before exporting.

The dataflow file does not include the following information:

  • The values for any parameters used in the dataflow. If you upload the dataflow, you must assign new values to the parameters used in the dataflow.
  • Parameters used in the export operation. If the original batch dataflow exports to a user-defined target or a customized export folder, you cannot obtain the target and export folder information from the dataflow file. If you run the uploaded dataflow, it exports to a default folder at the default target location. You must customize the export operation in the uploaded batch dataflow so that the dataflow exports the result to the desired location.
  • Schedule defined for the dataflow. After you upload the dataflow, to run it regularly, you must re-define the schedule. The schedule previously associated with the original dataflow is not retained during the download.
  • Export target and any associated export UDF. Before you run the dataflow, you will need to verify that a specified export target exists on a new cluster. In addition, you will need to confirm that any export UDF needed for export to the export target is configured on your cluster.

Uploading

You can upload a batch dataflow that was downloaded from the same Xcalar cluster or any other Xcalar cluster. If the dataflow file has one or more UDFs associated with it (for example, a UDF is used in one of the operations in the dataflow), the UDFs of the cluster are affected in one of the following ways:

  • A batch dataflow's UDFs are uploaded with the dataflow and are local to the dataflow. Uploading does not impact other UDFs used for modeling or dataflows.
  • If a UDF used by the batch dataflow is in the default module, instead, the cluster uses the UDF in its default module when running the batch dataflow. Uploading a dataflow does not overwrite the Default module for the cluster.

To upload a dataflow file, follow these steps:

  1. Click the icon for uploading a dataflow file.
  2. In the Upload Dataflow window, specify the following information:
    • The location of the dataflow file, which must be in the .tar.gz format. You can click BROWSE to navigate to the location and select the file.
    • Name of the dataflow if you want to change the name.
  3. Click UPLOAD.

Saving or displaying a batch dataflow graph as an image file

You can save a batch dataflow graph as an image (for example, to include the image in a presentation or to share the image with others). The image is saved in the PNG format at the download location, which is determined by your browser setting.

Alternatively, you can display the dataflow graph as a PNG image in a new tab of your browser.

Simply right click anywhere in the batch dataflow graph to display a menu. The menu options enable you to save the image or display the image in a new browser tab.
