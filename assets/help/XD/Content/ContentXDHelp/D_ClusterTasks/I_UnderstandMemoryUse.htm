Understanding the memory usage

The Xcalar architecture makes optimal use of system resources (such as memory, CPU, and network resources). Xcalar can optimize memory usage even on a heterogeneous cluster where server nodes have different amounts of memory installed.

About Xcalar Compute Engine's use of the memory hierarchy

Xcalar Compute Engine's distributed shared memory (DSM) architecture automatically uses and manages the tiers of a memory hierarchy, which are shown in the following list:

  • DRAM
  • Storage Class Memory (SCM)
  • Solid state drives (SSDs)
  • Hard disk drive (HDD)

Xcalar Compute Engine uses sophisticated policies to optimally distribute data and meta data across the cluster. Xcalar also gives you fine-grained control over the distribution of data and meta data across the cluster. Such control is particularly useful when you have to deal with non-uniform data distribution or correlation that occurs in real-world applications. Production systems often need to run optimized dataflows in a similar way as database administrators optimize SQL queries. Contact Xcalar if you need Xcalar's Field Technology Engineers to help operationalize dataflows for you or provide training on operationalizing dataflows.

Xcalar Compute Engine also uses demand paging to move data between main memory and lower tiers of the memory hierarchy. When the Xcalar cluster is low on memory, Xcalar writes data to lower tiers of the memory hierarchy. This process is called serialization. When more memory is available, Xcalar moves data from the demand paging area back to memory. This process is called deserialization.

While serialization and deserialization take place without user intervention, Xcalar recommends that users conserve memory so that Xcalar Compute Engine can deliver the highest possible performance.

Best practices for conserving memory

Xcalar Design users can follow these best practices to conserve memory:

  • Reduce the number of result sets as described in Effect of a result set on memory consumption.

  • Reduce the number of fields in the Data Browser as described in Understanding the Project operation.

Xcalar is designed to automatically drop the result sets after certain period of inactivity. The result set can be regenerated by right clicking on a dataflow Node and selecting the Execute menu item. You can prevent the system from dropping a result set by right clicking on the dataflow Node and selecting Lock Result menu item.

What happens when the cluster is low on memory

If the cluster is healthy without low-memory issues, a green icon is displayed next to the user icon in the top right corner of Xcalar Design. You can mouse over this icon to view the percentage of memory used. Click the icon to display the Statistics window, where you can see more details about resource usage. If Xcalar Compute Engine is low on memory, Xcalar Design displays a warning icon next to the user icon. A yellow icon indicates that memory consumption is at 60% and a red icon indicates that memory consumption is at 80%. A warning tooltip is also displayed, recommending the action to take for releasing memory.

If you have any result sets that can be dropped, clicking the warning icon displays the Drop Result Sets modal window otherwise it displays the Datasets panel, where you can deactivate any dataset that you are not using. For more information on how to drop a result set, see Effect of a result set on memory consumption.

To deactivate a table, click the Sources icon and then theTables icon in the sidebar. It displays the Tables panel, where you can right click on a table name and deactivate it.
