Xcalar SQL Limitations and Restrictions

This section describes the limitations and restrictions when working with Xcalar SQL. For a list of SQL ANSI standard features and subfeatures and the support level Xcalar provides for each feature, see Xcalar SQL compliance.

Window function restrictions:

The following window functions are supported without the OVER clause:

  • Values - FIRST_VALUE(), LAST_VALUE(), LAG(), LEAD()
  • Ranking - CUME_DIST(), DENSE_RANK(), PERCENT_RANK(), ROW_NUMBER()

IMPORTANT: Window functions depend on sort ordering specified by the ORDER BY clause, which is part of the OVER clause. If you do not include the OVER clause when using the ORDER BY clause, the results are non-deterministic.
TIP: Functions, such as LAG, LAST, LEAD, FIRST, and RANK do not persist the order after a calculation. If you do not want the inherent sort order result, Xcalar recommends adding another ORDER By clause outside of the ORDER BY clause at the end of a statement.

Frame clause restrictions:

In the frame clause of a window function, you must specify where the frame starts and ends.

The Xcalar SQL default frame clause includes all the row values from the start of the partition through to the current row of the partition including all the rows that have the same values as the current row:

RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW

The frame clause requires the OVER () clause and must be associated with the ORDER BY clause.

Where:

  • UNBOUNDED PRECEDING is where the frame starts with the first row of the partition.
  • CURRENT ROW is where the frame ends with the current row or last value of the current row based on the ORDER BY clause.

Operations not supported

The following operations are not supported:

  • SQL queries that run on less than one table
  • Counting values in multiple columns with the COUNT function.
    • TIP: Include the GROUP BY clause. For example, the following removes duplications and retrieves all possible column combinations from three columns before counting the values:

    SELECT COUNT (*) FROM (SELECT COUNT(*) FROM table GROUP BY col1, col2, col3)

  • Regular expressions, such as REGEXP_SUBSTR.
  • Xcalar SQL does not have a default sort order, users must include the ORDER BY clause if sorting is required.
  • Timezones in the TIMESTAMP function.

Differences between Xcalar SQL and Oracle SQL

The following list describes a few differences between Oracle and Xcalar SQL and how to achieve with Xcalar SQL the same or similar results as Oracle SQL:

  • Xcalar SQL does not support Oracle pseudocolumns.
    • NOTE: The Xcalar SQL ROW_NUMBER function is similar to the Oracle SQL ROWID pseudocolumn function. For example, SELECT ROW_NUMBER()FROM table
  • In Xcalar SQL the padding character argument is mandatory for both LPAD and RPAD functions. Example syntax,
    • Oracle SQL: LPAD(string, length [,pad])
    • Xcalar SQL: LPAD(string, length, pad)

      • where,

      • string, is the original text string value. if the length of the original text is larger than the length parameter, extra characters are removed.
      • length, is the length of the text string after the padding value is included.
      • pad, is the character padding value.
  • Xcalar SQL does not support character removal with the LTRIM or RTRIM functions, the LTRIM and RTRIM functions only remove the whitespace character. Example syntax,
    • Oracle SQL: LTRIM(string [,trim_string])
    • Xcalar SQL: LTRIM(string)

      • where,

      • string, is the character value that you want trimmed.
      • trim_string, is the value of the trimmed characters.
  • You can only apply the Xcalar SQL TRUNC function on TIMESTAMP data types. The Xcalar SQL TRUNC function also requires both date and format arguments. Example syntax,
    • Oracle SQL: TRUNC(date[,fmt])
    • Xcalar SQL: TRUNC(date, fmt)

      • where,

      • date, is the date value that you want truncated.
      • fmt, is the unit of measure for truncating.
    TIP: Xcalar recommends using a user-defined function (UDF) for other data types.
    NOTE: The Xcalar SQL ROUND() function is similar to the Oracle SQL TRUNC(number) function.
    NOTE: Xcalar SQL supports the ISO 8601 Date format.
  • Xcalar SQL does not support the Oracle SQL MEDIAN function, which returns the middle value from numeric or datetime data type values.
    • TIP: Xcalar recommends using the Dataflow Mode in Xcalar Design.
  • Xcalar SQL does not support the Oracle SQL TO_CHAR function that converts a DATE or a NUMBER data type into a displayable text string.
    • TIP: Xcalar recommends using the STRING() function without an argument.
    NOTE: Xcalar does support the CHAR function.
  • Xcalar SQL does not support the Oracle SQL TO_NUMBER function that converts other data types to an integer data type.
    • TIP: Xcalar recommends using the Xcalar SQL INT(), FLOAT(), or MONEY() functions without an argument.
  • The Oracle SQL COALESCE function requires at least two arguments, whereas Xcalar SQL supports one or more arguments.
